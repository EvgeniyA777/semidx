# Working Note: Codex Hybrid Exploration Policy

This note captures a practical answer to the question: should this repository rely more on CLI/skills than on SCI MCP for agent exploration?

This note is policy guidance only. It does not itself require changing `AGENTS.md`.

## Short Answer

Do not replace MCP with CLI.

Do not keep a rigid MCP-only first pass either.

The right operating mode for this repository is:

`MCP-primary, CLI-constrained`

Meaning:

- use SCI MCP for semantic narrowing, ranking, and compact context selection
- use CLI and filesystem tools for targeted follow-up only after the search space is already narrowed
- do not allow broad raw file reads to replace semantic retrieval, because that quickly burns context on large files

## Why This Repository Needs a Hybrid Policy

Two bad extremes already exist:

1. **Strict MCP-everywhere**

- too many sequential tool calls for simple structural questions
- unnecessary latency before seeing code
- brittle when language activation or semantic retrieval is temporarily misaligned with the task

2. **CLI-everywhere**

- `rg` + raw reads drift into large-file inspection
- files like `runtime/adapters.clj` and other hotspots can consume tens of thousands of tokens quickly
- semantic ranking and index state are bypassed, so the agent loses the main advantage of SCI

This repo is too large and too semantically rich for either extreme.

## Observed Cost Model

The practical tradeoff is not only “which tool finds the answer”, but also “how many tokens each useful step costs”.

Approximate shape:

| Operation | Typical token cost | Call shape |
|---|---:|---|
| `repo_map` | low | compact MCP |
| `resolve_context` | low-to-medium | compact MCP |
| `expand_context` / `fetch_context_detail` | medium | compact MCP |
| full MCP staged flow | low-to-medium total, but sequential | 3-5 dependent calls |
| `rg -n` | low | 1 direct call |
| bounded `sed`/read | medium | 1 direct call |
| broad raw file read on a hotspot | very high | 1 direct but expensive call |

Implication:

- MCP is usually better for narrowing
- CLI is usually better for precise follow-up
- raw file reads are the dangerous part, not CLI itself

## Proposed Operating Policy

### Use MCP for

- first semantic narrowing of a task
- ranked relevance selection
- snapshot/selection-driven context work
- cases where the agent does not yet know which file actually matters

Canonical first pass:

- `create_index`
- `repo_map`
- `resolve_context`

Use `expand_context` and `fetch_context_detail` only when the returned selection is still too broad.

### Use CLI and filesystem tools for

- reading a file already identified by MCP or by an exact symbol/path match
- `rg -n` on a known directory, path, namespace, symbol, or fixture id
- tests, benchmarks, and runtime CLI verification
- exact diff inspection and git status checks

CLI is a follow-up tool here, not the discovery authority.

## Optional Tier 0 Helper

If an agent repeatedly drifts into large-file reads, a cheap optional helper is a one-shot file-size map before broader inspection.

Example:

```bash
find src test -type f \( -name '*.clj' -o -name '*.ex' -o -name '*.exs' -o -name '*.py' -o -name '*.ts' -o -name '*.java' \) -exec wc -l {} + | sort -rn
```

Why this can help:

- one command gives a current size map of the repo hotspots
- the agent can decide early which files must never be read whole
- it reduces the chance of spending tens of thousands of tokens on one accidental read

This should stay optional and tactical, not a required first step for every task.

## Required Guardrails

If this hybrid policy is adopted, the following guardrails are mandatory.

### 1. No broad raw reads before narrowing

Do not `cat`/`sed` large or unknown files before MCP or an exact path/symbol match has narrowed the target.

### 2. Read budgets

For direct code reads:

- default to bounded reads, roughly `120-200` lines at a time
- avoid more than `2-3` raw file reads in a row without another narrowing step
- if more than a few candidate files appear, return to MCP or a structured CLI summary instead of continuing to read files blindly

### 3. Prefer structured CLI over raw file output

If CLI is added or expanded, it should return compact summaries:

- unit ids
- symbols
- paths
- line numbers
- diagnostics
- benchmark summaries

It should not default to dumping full source text.

### 4. Treat broad filesystem exploration as a fallback, not a primary mode

Do not replace `resolve_context` with broad `rg` over the whole repo as the default search strategy.

Allowed:

- targeted `rg -n` on a known path, namespace, symbol, module, or fixture id
- bounded reads after narrowing

Discouraged:

- wide grep crawls as the first answer to every question
- repeated whole-file reads across multiple hotspot files

### 5. Keep semantic retrieval as the control plane

Even when CLI is used heavily, SCI MCP should remain the primary tool for:

- semantic discovery
- relevance ranking
- stateful context narrowing

## Recommended CLI Shape For This Repo

If the repository wants agent-oriented CLI support, add thin, structured commands instead of generic raw-output commands.

High-value commands:

- `index` summary
- `units --module ...`
- `callers --unit-id ...`
- `parse-file --path ... --engine ...`
- benchmark subset wrappers

Recommended property:

- JSON or similarly structured output
- compact, line-oriented summaries
- no large source dumps by default

## Complementary SCI Improvement

An SCI-side improvement that would strengthen this policy without changing agent behavior would be:

- add `line_count` or similar size metadata to `repo_map`

Why:

- the data is useful for read budgeting
- it would let an agent make better follow-up decisions without a separate filesystem scan

This is a runtime/MCP enhancement, not a policy change.

## Where Skills Help

A skill helps with:

- choosing the right tool for the job
- remembering canonical commands
- enforcing guardrails such as “MCP first for narrowing, CLI second for targeted follow-up”

A skill does **not** by itself add new capabilities.

The real value comes from the combination:

- repo policy
- compact CLI surfaces
- a skill that encodes the intended tool-selection behavior

Skill should not claim to replace SCI MCP. It should reinforce the hybrid policy.

## Recommended Integration Order

1. Keep SCI MCP as the semantic-discovery authority.
2. Add or standardize compact CLI surfaces for targeted semantic queries.
3. Add a Codex skill that documents the hybrid tool-selection policy and the canonical commands.
4. Optionally add line-count awareness through SCI or a tactical filesystem helper.
5. Only after that consider further relaxation of repo-level exploration rules.

## Decision

For this repository, the best policy is not:

- “MCP only”
- or “CLI instead of MCP”

It is:

`MCP for narrowing, CLI for precise follow-up, hard guardrails on raw reads`

That gives better speed without losing the token discipline and semantic advantages of SCI.

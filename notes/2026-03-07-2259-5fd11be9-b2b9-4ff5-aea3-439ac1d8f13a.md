# Working Note: Research — Semantic Code Indexing, repo-map, Stack Graphs

## О чём файл

Исследовательская заметка по идее из исходного текста [../Semantic Code Indexing.md](../Semantic%20Code%20Indexing.md):
- сначала идея была ошибочно прочитана как запрос на вычитку файла;
- затем пользователь уточнил, что задача не редактировать текст, а оценить саму идею;
- после этого из текста были извлечены внешние проекты и подходы, которые уже делают близкие вещи;
- затем по первоисточникам были собраны ключевые рабочие идеи для `aider/repo-map` и `GitHub Stack Graphs`.

Заметка сохраняет логику текущего чата полностью и фиксирует доказательную трассу проверки.

## Что подтверждено

- Исходная идея сильна как `structure-aware code retrieval`, но переобещает, если называть ее полноценным `semantic indexing`.
- В исходном тексте как наиболее близкие внешние системы упомянуты `aider / repo-map` и `GitHub Stack Graphs`.
- `Tree-sitter`, `ctags-style tags`, `clj-kondo` и `LSP` в тексте выступают скорее как строительные блоки, а не как готовые end-to-end системы.
- `aider/repo-map` полезен прежде всего как `context compressor + relevance ranker`, а не как глубокий семантический анализатор.
- `GitHub Stack Graphs` полезен прежде всего как инкрементальный и декларативный слой `name resolution`, а не как замена компилятору.
- Исходный файл не редактировался; работа в этой сессии была аналитической.

## Логика текущего чата

1. Пользователь дал команду `Вычитай файл`.
2. Так как в корне репозитория был только один подходящий markdown-файл, была проверена текущая папка и найден `Semantic Code Indexing.md`.
3. До внесения правок пользователь резко уточнил, что редактирование не требуется и задача — именно оценить идею.
4. Идея была оценена как практичная для навигации по репозиторию и экономии контекста, но как неполная для настоящего `semantic analysis`.
5. Затем пользователь попросил вытащить из текста сторонние проекты, которые уже делают что-то похожее.
6. Из исходного текста были выделены:
   - `aider / repo-map`
   - `GitHub Stack Graphs`
7. После этого пользователь попросил вытащить по ним ключевые рабочие идеи.
8. Для этого были проверены первоисточники `aider` и `GitHub`, после чего была собрана прикладная выжимка.
9. Затем пользователь попросил задокументировать весь этот ресерч в `notes` по канону workspace-notes, но с телом, заточенным именно под исследовательскую заметку.

## Выжимка по сторонним проектам

### `aider / repo-map`

- Не тащить весь репозиторий в контекст; в чат отправлять только нужные файлы и компактную карту остального репозитория.
- Карта репозитория должна хранить не полный код, а `file -> key symbols -> signatures / critical lines`.
- Карта должна подстраиваться под текущий чат, а не быть статическим индексом.
- Размер карты должен ограничиваться token budget.
- В карту должны попадать не все символы подряд, а наиболее важные и наиболее часто цитируемые/связанные.
- По исходнику `aider/repomap.py` видно, что практическая реализация опирается на:
  - извлечение `definitions` и `references`;
  - граф связей между файлами;
  - ранжирование через `PageRank`;
  - усиление веса для уже открытых файлов, упомянутых путей и идентификаторов;
  - подбор объема карты под бюджет токенов;
  - кэширование по `mtime`;
  - местами fallback через `pygments`, если Tree-sitter не дал достаточных ссылок.

### `GitHub Stack Graphs`

- Анализировать каждый файл изолированно на этапе индексации и сохранять из него локальные факты.
- На этапе запроса объединять локальные графы в граф репозитория/коммита.
- Кодировать правила связывания имен декларативно: `scope`, `shadowing`, `imports`, разрешение символов между файлами.
- Разрешать ссылки как поиск пути от `reference node` к `definition node` в объединенном графе.
- Для фильтрации ложных путей учитывать не только ребра графа, но и `symbol stack` / `precedence`.
- Использовать Tree-sitter как синтаксическую основу и отдельный DSL/слой для построения графа.
- Практическая ценность подхода в том, что он не требует полного build pipeline или глубокой интеграции с компилятором.
- Сам репозиторий `github/stack-graphs` архивирован и read-only с `2025-09-09`, но архитектурные идеи остаются полезными.

## Что это значит для дизайна текущей идеи

- Реалистичное позиционирование: не `full semantic code indexing`, а `structure-aware retrieval + symbol graph`.
- Для MVP стоит сначала строить:
  - иерархию файлов и модулей;
  - определения;
  - импорты/depends-on связи;
  - docstrings/comments;
  - явные call sites;
  - spans исходного кода для точечного fetch.
- Настоящую глубокую семантику лучше считать отдельным, более дорогим слоем.
- Практически ценная комбинация для MVP:
  - `repo map` для общей карты;
  - `skeletonization` для компрессии контекста;
  - `symbol graph` для навигации и relevance ranking;
  - точечная подгрузка тел функций/методов только по запросу.

## Трасса проверки

### Проверка 1. Что есть в текущем репозитории

- **Что было проверено:** содержимое корня текущего репозитория (`<repo-root>`)
- **Наблюдение:** в корне находился только один подходящий markdown-файл: `Semantic Code Indexing.md`; папки `notes/` не было
- **Вывод:** исходный объект анализа один, а notes-инфраструктуру нужно создать в репозитории вручную

### Проверка 2. Что именно упомянуто в исходном тексте

- **Что было проверено:** содержимое `Semantic Code Indexing.md`
- **Наблюдение:** как близкие внешние примеры прямо упомянуты `aider (repo-map)` и `GitHub Stack Graphs`; также упомянуты `Tree-sitter`, `ctags-style`, `clj-kondo`, `LSP`, `Datascript`
- **Вывод:** для ответа на вопрос `кто это уже делает` нужно отделять готовые системы от инфраструктурных инструментов

### Проверка 3. Канон заметок для workspace / notes

- **Что было проверено:** `<workspace-notes-root>/README.md`, `<skeletonizer-root>/notes/_template.md`, `<skeletonizer-root>/notes/README.md`
- **Наблюдение:** устойчивый канон одинаковый: `notes/`, имя файла `YYYY-MM-DD-HHmm-UUID.md`, frontmatter с `file_type`, `topic`, `created_at`, `author`, `language`; для аналитики обязательна секция `Трасса проверки`
- **Вывод:** в текущем репозитории можно сохранить мета-информацию и именование, а тело адаптировать под research-note

### Проверка 4. `aider / repo-map`

- **Что было проверено:** официальные docs `aider` и исходник `aider/repomap.py`
- **Наблюдение:** `repo-map` решает задачу компактной карты репозитория, работает под token budget, подстраивается под текущий чат и использует ранжирование графа связей для выбора наиболее полезного контекста
- **Вывод:** ключевая рабочая идея `aider` — не полнота, а максимальная полезность контекста на один токен

### Проверка 5. `GitHub Stack Graphs`

- **Что было проверено:** официальный блог GitHub и репозиторий `github/stack-graphs`
- **Наблюдение:** подход основан на пофайловом извлечении фактов и последующем разрешении ссылок через path search по объединенному графу; репозиторий архивирован
- **Вывод:** ключевая рабочая идея Stack Graphs — инкрементальный декларативный `name resolution`, а не full-program analysis

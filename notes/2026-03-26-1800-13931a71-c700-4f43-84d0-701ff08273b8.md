# Working Note: Elixir Tree-Sitter Readiness Status

This artifact closes the current Elixir tree-sitter tranche with an explicit readiness status instead of an implicit assumption.

## Delivered in This Tranche

- Stage 0 baseline harness landed
- Stage 1 Elixir boundary extraction landed
- Stage 2 Elixir tree-sitter structural extraction landed

## Current Readiness

Status:

`not-ready-for-default-switch`

Current decision:

- keep Elixir tree-sitter as an optional path
- do not flip default parser yet

## Green Evidence

- plain `clojure -M:test` remains green after the Stage 2 structural extraction
- Elixir regex benchmark subset still matches the recorded Stage 0 baseline
- Elixir now has a dedicated tree-sitter provider instead of a validation-only wrapper over the full regex provider result

## Open Blockers

1. Shared tree-sitter parser-path verification is still red under env-enabled tests.

Observed shape:

- `tree-sitter-parser-path-test` still fails across existing tree-sitter lanes (`clojure`, `elixir`, `java`, `typescript`)
- the failure is in the shared CLI/config invocation path, not only in the Elixir provider
- this must be resolved before Elixir default-switch work is credible

2. Sandbox-level HTTP bind restrictions still prevent the env-enabled full suite from serving as a final gate here.

Observed shape:

- env-enabled `clojure -M:test` hits `Operation not permitted` in HTTP server tests
- this is environmental, but it still means the full readiness gate was not closed in this workspace

3. Elixir benchmark subset is still red.

Current baseline remains:

- `retrieval_elixir_aliased_local_ambiguity_001`
- `retrieval_elixir_mixed_def_forms_001`
- `retrieval_elixir_exunit_module_scenario_001`

These failures predate the Stage 2 extraction and therefore remain baseline debt, not proof that the new provider regressed.

## Next Required Work Before Any Default Flip

- fix the shared tree-sitter CLI/config invocation until parser-path tests go green under env-enabled execution
- rerun env-enabled Elixir parity and shared parser-path tests in a workspace where HTTP bind tests can run
- only then decide whether a real Stage 3 parity/hardening tranche is sufficient for a later default-switch commit

## Decision Record

The current rollout stops here.

The next Elixir tree-sitter commit should be either:

- a shared tree-sitter invocation fix that clears parser-path verification, or
- a focused Elixir parity tranche once that shared blocker is gone

# Working Note: Roadmap Status Audit After Quality-Loop Foundations

Superseded by the later canonical status in `docs/roadmap-status.md` and the current operational snapshot in `MEMORY.md`. This note remains as a historical checkpoint from earlier on 2026-03-10.

Everything below should be read as a historical snapshot from before the later Phase 3/Phase 4 work landed; it is intentionally not the current source of truth for repository status.

## О чём файл

Заметка фиксирует сверку исходного roadmap "Product-Ready Roadmap After Quality-Loop Foundations" с текущим состоянием репозитория.

Отдельно зафиксировано:

- что уже реализовано
- что осталось не реализованным
- где в репозитории теперь лежит canonical short checklist
- почему checklist положен именно туда

## Что подтверждено

- Phase 2 больше не является открытым хвостом: lifecycle registry, governed replay comparison, scorecards, promotion gates, protected replay cases, `shadow-review`, и registry-backed policy selection на library/HTTP/gRPC/MCP уже реализованы.
- Старые формулировки "HTTP/gRPC/MCP ещё не умеют принимать policy selection input" и "protected replay datasets и shadow-vs-active только на library/eval слое" устарели.
- Clojure как главный язык Phase 3 продвинут существенно дальше MVP baseline: alias-aware calls, test-to-source linkage, dispatch-aware multimethod identity/ranking, macro-aware boundaries, recursive macro-generated ownership.
- После этого основным незакрытым хвостом roadmap остаются:
  - Elixir / Java / Python semantic-core deepening
  - language-aware capabilities for confidence and guardrails
  - большая часть Phase 4 hardening
  - почти весь Phase 5 self-improvement loop

## Что обновлено в репозитории

- `MEMORY.md` обновлён под актуальный roadmap tail, без уже закрытых Phase 2 пробелов.
- Создан `docs/roadmap-status.md` как canonical in-repo checklist.
- `README.md` теперь ссылается на `docs/roadmap-status.md`.

## Почему checklist положен в `docs/roadmap-status.md`

По каноническим публичным проектам наблюдается устойчивый паттерн:

- roadmap/status либо ведутся в отдельном официальном tracking artifact вне дерева кода
- либо держатся как один канонический governance/tracking артефакт, а не размазываются по случайным заметкам

Для этого репозитория нет отдельного публичного roadmap tracker как source of truth, поэтому ближайший канонический аналог внутри репозитория — одна versioned status page в `docs/`, а не:

- временная заметка в `notes/`
- случайный фрагмент в `README`
- только `MEMORY.md`, который предназначен для оперативной памяти, а не для внешне-читаемого roadmap status

Это вывод по аналогии с официальными проектами, а не буквальное требование одного конкретного стандарта.

## Короткий статус roadmap

- `Done`: Phase 2 governed quality loop и runtime policy selection across surfaces
- `Partial`: Clojure semantic-core, Elixir baseline, Java baseline, Python baseline, Phase 4 foundations
- `Not done`: language-aware capabilities, большая часть Phase 4 hardening, весь настоящий Phase 5 loop

## Трасса проверки

### Проверка 1. Структура текущего репозитория

- **Что было проверено:** дерево файлов и директорий репозитория
- **Наблюдение:** в проекте уже есть устойчивые canonical buckets `docs/`, `adr/`, `notes/`, `README.md`, `MEMORY.md`
- **Вывод:** roadmap status логично хранить либо в `docs/`, либо только во внешнем tracker; `notes/` подходит лишь для dated context

### Проверка 2. Реальный статус реализации по коду и тестам

- **Что было проверено:** `src/`, `test/`, `README.md`, `MEMORY.md`, `docs/`
- **Наблюдение:** policy registry, scorecards, `compare-policies`, `promote-policy`, `shadow-review`, protected cases, HTTP/gRPC/MCP selector-based policy lookup уже есть; Clojure semantic-core тоже заметно углублён
- **Вывод:** исходный roadmap теперь частично устарел и требует отдельного status artifact, а не только memory note

### Проверка 3. Как канонические проекты держат roadmap/status artifacts

- **Что было проверено:** официальные roadmap/tracking источники GitHub, Kubernetes, Go, Rust
- **Наблюдение:** крупные проекты держат roadmap/status либо в отдельном canonical tracker/repository, либо в issue-based governance artifacts с явным source of truth
- **Вывод:** внутри этого репозитория лучшая аналогия — один canonical file в `docs/`, а не разрозненные ad hoc заметки

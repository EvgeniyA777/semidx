# Working Note: Additive Staged Semantic Diff And Projection Rollout

Adopt additive staged rollout for new semantic diff / projection capabilities. Stabilize the core shape in library-internal APIs first, then expose additive public fields/tools without breaking existing behavior, and bring MCP, HTTP, and gRPC surfaces online stage by stage once snapshot semantics and result shapes are proven.

## Review Follow-Up: Reproduced Defects On `dev`

The rollout plan is functionally complete, but branch review found four defects that should be treated as follow-up fixes rather than deferred product ideas.

### 1. `literal_file_slice` violates `snapshot_id` semantics on current-snapshot reads

- Current behavior: when reading without `selection_id`, the implementation validates that the requested `snapshot_id` matches the in-memory index, then reads the file from disk.
- Reproduced failure: index file -> mutate file on disk without reindex -> request `literal_file_slice` with the old `snapshot_id` -> returned content reflects the live file, not the indexed snapshot.
- Intended contract: if a caller supplies `snapshot_id`, the returned text must come from that exact snapshot or fail explicitly.

### 2. `snapshot_diff` can misclassify unrelated add/remove pairs as `moved_or_renamed`

- Current behavior: rename pairing is driven by `semantic_fingerprint` equality.
- Reproduced failure: baseline contains one helper function, current contains a different helper with the same implementation shape; diff reports `moved_or_renamed` instead of `removed` + `added`.
- Intended contract: `moved_or_renamed` should require stronger evidence that the unit is truly the same semantic entity, not merely similar implementation.

### 3. Advisory semantic-quality runner can hide real execution failures behind stale output

- Current behavior: `scripts/run-semantic-quality-report.sh` checks only whether `OUT_PATH` exists and is non-empty after the CLI exits.
- Reproduced failure: pre-seed `OUT_PATH` with an old report -> run the script with a broken dataset path -> CLI fails, but the script exits `0`, emits summary output from the stale report, and marks the failure as advisory.
- Intended contract: execution failure must never be downgraded to advisory by reusing stale artifacts.

### 4. CI workflow artifact paths are inconsistent with the runner's `TMPDIR` behavior

- Current behavior: the runner writes to `${TMPDIR:-.tmp}`, while the GitHub Actions workflow publishes only `.tmp/semantic-quality-report.json` and `.tmp/semantic-quality-report-summary.md`.
- Reproduced failure: run the script with `TMPDIR` set to a custom directory -> report and summary are written there, while `.tmp/...` remains empty.
- Intended contract: CI should publish the artifacts that the runner actually produced, regardless of whether `TMPDIR` is set.

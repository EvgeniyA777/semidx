# Working Note: Elixir Tree-Sitter Implementation Slice

This note materializes the `ADR-028` follow-up for the highest-priority next parser-deepening lane: `elixir`.

It defines the bounded first implementation slice so the next code stage can be executed without reopening parser-strategy decisions.

## Goal

Add an optional Elixir tree-sitter parser path that improves structural extraction for ordinary Elixir code while preserving the current adapter parser as a bounded fallback.

The first slice is not a full semantic rewrite. It is a parser-strengthening tranche for the surfaces that already matter to retrieval quality in this repository.

## Parser Strategy

- Add an Elixir tree-sitter path as an additive parser engine.
- Keep the current Elixir adapter parser as the fallback path.
- Do not switch Elixir to tree-sitter-by-default in the same implementation stage.
- Reuse the existing diagnostics pattern already used by `clojure`, `java`, and `typescript`:
  - `tree_sitter_active`
  - `tree_sitter_unavailable`
  - `tree_sitter_missing_grammar`
  - `tree_sitter_parse_failed`
  - `tree_sitter_no_units`

## Public and Runtime Contract

- Add Elixir parser-engine selection following the existing `*_engine` shape.
- Add Elixir grammar lookup through the shared tree-sitter grammar wiring.
- Keep `parse-file` output shape unchanged:
  - `:language`
  - `:module`
  - `:imports`
  - `:units`
  - `:diagnostics`
  - `:parser_mode`
- Keep bounded fallback behavior: parser failure must degrade to the current Elixir adapter output, not to crash behavior.

## First-Slice Semantic Coverage

The first Elixir tree-sitter tranche must cover these already-important repository surfaces:

- `defmodule`
- `def`, `defp`, `defmacro`, `defmacrop`, `defdelegate`
- `test "..." do`
- `alias`, including brace aliases already covered by regression tests
- `import`
- `use`
- unqualified imported/use-expanded calls
- pipeline calls such as `|> normalize()`
- local captures such as `&normalize/1`
- `__MODULE__.foo(...)`
- `__MODULE__.Nested.foo(...)`
- nested modules as unit/module owners
- arity-aware caller resolution parity with the current adapter expectations

## Explicit Non-Goals for the First Slice

- no attempt to model full macro expansion
- no attempt to emulate the Elixir compiler
- no default-parser flip
- no confidence-ceiling bump
- no broad retrieval-policy changes

## Test and Acceptance Matrix

The code stage for this slice should not land unless all of the following are true:

1. Existing Elixir runtime regressions still pass.
2. Tree-sitter-specific Elixir tests prove parity for:
   - alias resolution
   - `use`-expanded imports
   - local-over-import precedence
   - arity-aware same-name shadowing
   - `defdelegate`
   - pipeline / `with` / capture handling
   - nested `__MODULE__` qualification
3. Fallback behavior is provably bounded when:
   - tree-sitter CLI is unavailable
   - Elixir grammar path is missing
   - parse fails
4. Public diagnostics remain explainable and aligned with existing tree-sitter lanes.

## Stage Exit Criteria

The implementation stage that consumes this note should end with:

- Elixir tree-sitter parser path implemented
- targeted runtime parity tests green
- full `clojure -M:test` green
- docs updated if new public parser options are exposed
- one dedicated commit
- one dedicated push

## Relationship to Other Decisions

- `ADR-028` defines why Elixir is the highest-priority next tree-sitter lane.
- Python parser deepening stays separate and should not be mixed into the same implementation stage.
- Lua remains on bounded regex-first parsing until its supported surface expands materially.

Для создания системы, которая строит «каркас» (скелет) проекта без загрузки всего кода в контекст, эффективнее всего использовать подход **Semantic Сode Indexing** на базе **Tree-sitter** и **графовых структур**.

Вместо того чтобы скармливать LLM сырой текст, мы превращаем проект в дерево определений и ссылок.

---

## **1\. Архитектура системы: «LSP на стероидах»**

Суть в том, чтобы извлечь только **сигнатуры** (API) и **структуру вложенности**, игнорируя реализацию функций.

## **Шаг 1: Многоязычный парсинг (Tree-sitter)**

Для каждого файла в проекте запускается Tree-sitter с соответствующим грамматическим модулем. Вам нужно извлекать конкретные типы узлов (S-выражения):

* **Определения:** function\_definition, class\_definition, method\_definition, variable\_declaration.  
* **Метаданные:** Имя сущности, параметры, возвращаемый тип и *docstrings*.

## **Шаг 2: Создание «Скелета» (Skeletonization)**

Вы генерируете упрощенную версию файла.

* **Сырой код:** 100 строк реализации сложного алгоритма.  
* **Скелет:** defn process-data \[ids\] \-\> result \+ краткий комментарий из докстринга.  
  Это сжимает объем данных в 10–50 раз, сохраняя семантику для LLM.

## **Шаг 3: Построение графа зависимостей (Symbol Graph)**

Чтобы система «видела» связи между файлами, нужно сопоставить использование (call sites) с определениями. Здесь помогают:

1. **Tags (ctags-style):** Простое сопоставление имен.  
2. **Stack Graphs:** Технология от GitHub (на базе Tree-sitter), которая позволяет инкрементально разрешать области видимости (scope) без полной компиляции.

---

## **2\. Алгоритм работы при запросе пользователя**

Когда пользователь задает вопрос по проекту, система работает как поисковик:

1. **Map-Phase:** LLM получает «карту высот» — список всех файлов и их основных классов/модулей (всего пара сотен токенов).  
2. **Identify-Phase:** LLM выбирает, какие модули ей кажутся релевантными.  
3. **Fetch-Phase:** Система подтягивает из индекса **только скелеты** выбранных модулей.  
4. **Final Prompt:** В окно контекста попадает:  
   * Глобальная структура (дерево папок).  
   * Детальные сигнатуры релевантных файлов.  
   * *Только один* конкретный кусок кода, который нужно изменить.

## **3\. Инструментарий для реализации**

Если вы планируете строить это на **Clojure/Babashka** (учитывая ваши интересы), архитектура может выглядеть так:

| Компонент | Технология | Роль |
| :---- | :---- | :---- |
| **Parser** | tree-sitter-clojure / clj-kondo | Извлечение фактов о коде. |
| **Storage** | **Datascript** (In-memory Datalog) | Идеально подходит для хранения графа связей между функциями и файлами. |
| **Protocol** | **LSP (Language Server Protocol)** | Можно использовать готовые серверы для получения символов проекта. |
| **Orchestrator** | Ваша многоагентная система | Один агент ищет нужные файлы, другой анализирует их скелеты. |

## **Почему это лучше классического RAG?**

Обычный RAG (векторный поиск) часто разрывает контекст: он может найти определение функции, но не увидит, что она находится внутри важного класса или зависит от глобального конфига. **Tree-sitter \+ Skeletonization** сохраняет иерархическую логику кода.
